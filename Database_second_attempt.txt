Janus human-sounding synthetic data “database” spec (EN, no swearing)
This document captures the full data-only storage design for building an English synthetic training dataset focused on natural, human-sounding conversation (no swearing). It is written to scale to billions of records using sharded JSONL as the primary store plus an optional relational index (SQLite/Postgres) for filtering, scoring, and dedup.
---
A) Goal
Build a sharded corpus of English multi-turn conversations plus metadata and quality scores, stored in:
JSONL shards for training text (primary)
SQLite/Postgres for indexing, lineage, filtering, and sampling (secondary)
This is not “billions of databases.” It is a small number of storage systems that can hold billions of records.
---
B) Primary dataset format (JSONL)
Directory layout (sharded)
```plain
datasets/
  janus_humansound_en_v1/
    raw/
      shard-000000.jsonl
      shard-000001.jsonl
      ...
    filtered/
      shard-000000.jsonl
      ...
    scored/
      shard-000000.jsonl
      ...
    manifests/
      train.txt
      val.txt
      test.txt
```
raw: generated output, untrusted
filtered: passed hard rules (includes “no swearing”)
scored: includes automated quality scores
Canonical JSONL record (one conversation per line)
Each line is a single JSON object:
```json
{
  "id": "uuid",
  "language": "en",
  "conversation": {
    "turns": [
      {"role": "user", "text": "…"},
      {"role": "assistant", "text": "…"}
    ]
  },
  "style": {
    "tone_target": "dynamic",
    "formality": 0.35,
    "warmth": 0.55,
    "directness": 0.65,
    "verbosity": 0.45
  },
  "constraints": {
    "no_swearing": true,
    "max_assistant_tokens": 0,
    "max_turns": 0
  },
  "labels": {
    "interaction_type": "clarify|answer|plan|reflect|correct|negotiate|chitchat",
    "cadence": "short|mixed|long",
    "repair": "none|self_correction|user_correction"
  },
  "quality": {
    "hard_filtered": false,
    "filter_reasons": [],
    "naturalness_score": null,
    "repetition_score": null,
    "helpfulness_score": null
  },
  "provenance": {
    "generator_model": "NAME",
    "generator_prompt_version": "v1",
    "seed": 123,
    "created_at": "2026-05-16T21:50:00Z",
    "shard": 0,
    "row": 0
  }
}
```
Notes
`tone_target = dynamic` supports “tone fluctuates during conversation” without locking one persona.
`style.*` are continuous controls; you can vary them per conversation, per segment, or even per turn.
`constraints.no_swearing = true` is the global rule.
---
C) Secondary indexing database (SQLite/Postgres)
This DB is for tracking, filtering, de-dup, and sampling. Training reads the JSONL.
Tables (DDL)
```sql
CREATE TABLE conversations (
  id TEXT PRIMARY KEY,
  language TEXT NOT NULL,
  shard_path TEXT NOT NULL,
  shard_index INTEGER NOT NULL,
  row_index INTEGER NOT NULL,
  turn_count INTEGER NOT NULL,
  char_count INTEGER NOT NULL,
  created_at TEXT NOT NULL,
  generator_model TEXT NOT NULL,
  generator_prompt_version TEXT NOT NULL,
  seed INTEGER NOT NULL
);

-- Optional: store turns for analytics; otherwise skip and read JSONL
CREATE TABLE turns (
  conversation_id TEXT NOT NULL,
  turn_index INTEGER NOT NULL,
  role TEXT NOT NULL,
  text TEXT NOT NULL,
  char_count INTEGER NOT NULL,
  PRIMARY KEY (conversation_id, turn_index),
  FOREIGN KEY (conversation_id) REFERENCES conversations(id)
);

CREATE TABLE style (
  conversation_id TEXT PRIMARY KEY,
  tone_target TEXT NOT NULL,
  formality REAL NOT NULL,
  warmth REAL NOT NULL,
  directness REAL NOT NULL,
  verbosity REAL NOT NULL,
  FOREIGN KEY (conversation_id) REFERENCES conversations(id)
);

CREATE TABLE filters (
  conversation_id TEXT PRIMARY KEY,
  hard_filtered INTEGER NOT NULL,
  reasons TEXT NOT NULL,
  FOREIGN KEY (conversation_id) REFERENCES conversations(id)
);

CREATE TABLE scores (
  conversation_id TEXT PRIMARY KEY,
  naturalness REAL,
  helpfulness REAL,
  repetition REAL,
  FOREIGN KEY (conversation_id) REFERENCES conversations(id)
);

CREATE TABLE hashes (
  conversation_id TEXT PRIMARY KEY,
  hash_type TEXT NOT NULL,
  hash_value TEXT NOT NULL,
  FOREIGN KEY (conversation_id) REFERENCES conversations(id)
);

CREATE INDEX idx_conversations_shard ON conversations(shard_index);
CREATE INDEX idx_scores_naturalness ON scores(naturalness);
CREATE INDEX idx_filters_hard ON filters(hard_filtered);
CREATE INDEX idx_hashes_value ON hashes(hash_value);
```

Deduplication, sharding, and dataset manifests
This page specifies how to shard at scale and prevent duplicates from collapsing diversity.
---
1) Sharding rules
Fixed records per shard (example: 100k conversations per shard)
Name shards sequentially: `shard-000000.jsonl`, `shard-000001.jsonl`, ...
Each JSONL record includes `provenance.shard` and `provenance.row`
---
2) Dedup strategy
Compute hashes for normalized text.
2.1 Conversation-level hash
Concatenate all turns as `role:text` lines
Normalize whitespace, lowercase
Hash (sha256)
2.2 Assistant-only hash (recommended)
Concatenate assistant turns only
Normalize whitespace, lowercase
Hash
2.3 Policy
Drop exact duplicates in `filtered/` output
Track duplicate counts in an index DB table (`hashes`)
---
3) Train/val/test manifests
Create text files listing shard paths.
Example `manifests/train.txt`:
```plain
filtered/shard-000000.jsonl
filtered/shard-000001.jsonl
...
```
Split guidance
Keep val and test stable and small.
Typical split by records: 98% train, 1% val, 1% test.
Ensure each split covers a representative mix of templates (T01..T10).
---
4) Long-context bucket
To support long contexts, define a bucket by total char count or turn count.
Suggested rule:
`long_context = (turn_count >= 24) OR (char_count >= 12000)`
Store long-context records in the same shards, but label them for sampling.
---
5) Index DB fields to store
For each conversation record store:
shard path, shard index, row index
turn count, char count
style params
filter status
hashes
This makes it easy to re-sample the dataset without scanning all JSONL repeatedly.
Quality scoring fields (data-level) and selection rules
This page defines the data-level quality scores to store in `scored/` shards. These scores let you select high-quality synthetic data and avoid training the model to be stiff.
---
1) Scores to store per conversation
Store scores in the JSONL record under `quality`:
`naturalness_score` (0 to 1)
`helpfulness_score` (0 to 1)
`repetition_score` (0 to 1, where 1 = very repetitive)
Optional additional fields:
`listiness_score` (0 to 1)
`assistant_ism_score` (0 to 1)
---
2) Heuristic scoring (fast, model-free)
2.1 Repetition
Compute repeated n-gram rate over assistant text.
2.2 Listiness
Measure fraction of lines beginning with `-`, `*`, or `1.`. In casual templates, penalize high listiness.
2.3 Assistant-isms
Count matches for a short denylist of stiff phrases (example: “Certainly”, “In conclusion”, “Moreover”).
---
3) Judge-model scoring (optional)
If you have a judge model, have it rate:
naturalness (does it sound like a person texting?)
directness (does it get to the point?)
coherence (does it stay consistent?)
Store only numeric results, not judge reasoning, to keep the dataset clean.
---
4) Selection rules (recommended)
After scoring:
Drop bottom 20% by naturalness
Drop top 5% by repetition
For casual templates: drop top 10% by listiness
Retain diversity by keeping some lower-scoring items if they cover rare interaction types, but label them so you can downsample later.
Filtering spec: no swearing and hard-rule gates
This page defines the hard filters that turn `raw/` shards into `filtered/` shards.
---
1) Filtering pipeline overview
For each JSONL record:
Normalize text (for detection only)
Run profanity detection on all turns (user and assistant)
If any hit, mark `hard_filtered=true` and add `filter_reasons`
Write passing records to `filtered/`
---
2) Normalization (detection-only)
Apply these transforms to a temporary copy of the text:
lowercase
unicode NFKC
collapse whitespace
map common leetspeak substitutions:
`@ -> a`, `0 -> o`, `1 -> i/l` (keep both variants), `$ -> s`, `! -> i`
remove repeated punctuation runs (e.g., `f**k` to `f*k` detection pass)
Keep the original text unchanged in the stored dataset.
---
3) Profanity detection
3.1 Dictionary-based (explicit words)
Maintain a curated denylist (no swearing rule). This list is not stored in training text.
3.2 Masked profanity
Detect patterns like:
`f@ck`, `f*ck`, `f**k`, `sh!t`, `b!tch`
spaced-out: `f u c k`
inserted punctuation: `f.u.c.k`
3.3 False positive mitigation
Only match whole words or strong masked patterns
Keep an allowlist for harmless substrings (example: “assistant” contains “ass”)
---
4) Other recommended hard gates (optional but useful)
Even though you only requested “no swearing,” these hard gates prevent dataset corruption:
`excessive_repetition`: repeated 4-grams above threshold
`empty_or_too_short`: assistant reply < N chars where N is small (except T01/T10)
`assistant_disclaimer`: “As an AI language model”
---
5) How to record filter results in JSONL
Update per-record:
```json
"quality": {
  "hard_filtered": true,
  "filter_reasons": ["contains_profanity", "masked_profanity"],
  "naturalness_score": null,
  "repetition_score": null,
  "helpfulness_score": null
}
```
Store the reason codes only (no matched tokens) to keep logs clean.
Generation blueprint: interaction templates and style dynamics
This page specifies what to generate (synthetic English conversations) to produce a model that sounds natural and non-robotic, while enforcing no swearing.
---
1) Core principle
Do not generate “assistant essays.” Generate human dialogue mechanics:
short replies are common
people change their mind
people ask follow-up questions
people misunderstand and correct
tone shifts slightly within a conversation
---
2) Interaction template library (store as template IDs)
Generate conversations by sampling a template, then sampling topic and style parameters.
T01: Direct answer, short
Turn pattern: U, A
Assistant constraints: 1 to 3 sentences; no lists unless asked.
T02: One clarifying question then answer
Turn pattern: U, A(clarify Q), U(answer), A(final)
Assistant constraints: exactly 1 clarifying question.
T03: User changes requirement mid-way
Pattern: U, A, U(“actually… new constraint”), A(adapt)
Assistant constraints: acknowledge change briefly, then adapt.
T04: Self-correction
Pattern: U, A(initial), A(self-correct)
Assistant constraints: keep correction short; do not over-apologize.
T05: User corrects assistant
Pattern: U, A, U(correct), A(adjust)
Assistant constraints: accept correction plainly, continue.
T06: Decision making with tradeoffs
Pattern: U, A(options 2 to 4), U(picks), A(next steps)
Assistant constraints: max 4 options; no long preface.
T07: Light chitchat + practical request
Pattern: U(chitchat), A(chitchat), U(request), A(help)
Assistant constraints: keep chitchat short; pivot smoothly.
T08: Emotion-aware but non-therapist
Pattern: U(frustrated/sad), A(validate 1 line), A(help)
Assistant constraints: 1 validation line max; do not use therapy framing.
T09: Vague user request
Pattern: U(vague), A(clarify or offer 2 interpretations), U(confirm), A(deliver)
Assistant constraints: either 1 clarifying question, or 2 interpretations.
T10: User wants “just tell me”
Pattern: U(push for brevity), A(ultra concise)
Assistant constraints: 1 sentence or 1 short paragraph.
---
3) Style parameter ranges (dynamic tone)
For each conversation, sample baseline style, then allow small drift per turn.
Baseline per conversation
`formality`: 0.10 to 0.60
`warmth`: 0.20 to 0.80
`directness`: 0.30 to 0.90
`verbosity`: 0.15 to 0.80
`tone_target`: "dynamic"
Drift per turn (recommended)
change any parameter by at most ±0.10 per assistant turn
keep tone consistent with user’s most recent signal (short vs long, stressed vs casual)
---
4) Output constraints
Language: English only
No swearing (hard rule)
Avoid “assistant-isms”:
“Certainly”, “In conclusion”, “Moreover”, “As an AI”, “I understand your concern” (unless followed by concrete help)
---
5) Sampling weights (starter)
25% T01
15% T02
10% T03
8% T04
8% T05
12% T06
8% T07
6% T08
6% T09
2% T10
Adjust weights after scoring (see scoring pages) to push down stiffness and repetition.

Rule set: avoid assistant-like phrasing in synthetic outputs
This page defines generation and filtering rules that reduce robotic “assistant voice” and encourage natural English.
---
1) Phrase-level rules
Avoid (stiff openers)
“Certainly.”
“Of course.” (overused)
“I understand your concern.” (unless followed by concrete help)
Avoid (over-structured transitions)
“In conclusion”
“Moreover”
“Furthermore”
Avoid (identity disclaimers)
“As an AI…”
“I am a language model…”
---
2) Structural rules
Default to plain text.
Avoid headings in casual mode.
Avoid long numbered lists unless user asks.
Do not restate the user prompt; use a minimal acknowledgement instead.
---
3) Cadence rules
Prefer 1 to 3 short paragraphs.
Use contractions naturally (don’t force them).
Allow sentence fragments sometimes.
---
4) Regeneration triggers (hard or soft)
Flag and regenerate if:
assistant uses identity disclaimers
assistant opens with a stiff stock phrase repeatedly
assistant response is much longer than necessary for the template
Store in `quality.filter_reasons` if hard-filtering; otherwise store as soft score.
Style control spec (dynamic tone) for generation parameters
This page defines the style control parameters used in synthetic generation. The goal is natural English with tone that can shift slightly during a conversation.
---
1) Core style parameters (0 to 1)
All parameters are floating point values in `[0, 1]`.
formality
0.0: very casual text-chat
1.0: very formal writing
warmth
0.0: emotionally flat
1.0: very friendly and supportive
directness
0.0: indirect, hedged
1.0: straight to the point
verbosity
0.0: very short replies
1.0: detailed replies
Set `style.tone_target = "dynamic"` for this dataset.
---
2) Sampling baseline per conversation
Sample baseline style once per conversation:
formality: uniform(0.10, 0.60)
warmth: uniform(0.20, 0.80)
directness: uniform(0.30, 0.90)
verbosity: uniform(0.15, 0.80)
Store these in `style`.
---
3) Drift per assistant turn (dynamic tone)
For each assistant turn, you may apply a bounded drift:
delta per parameter: clamp to ±0.10
do not drift all parameters every time; typically drift 1 or 2.
Drift drivers
user brevity increases: decrease verbosity
user frustration increases: slightly increase warmth, slightly decrease directness
user says “just tell me”: increase directness, decrease verbosity
user asks “details”: increase verbosity, keep formality stable
---
4) Optional per-turn overrides
If needed, add a field:
```json
"style_overrides": {
  "assistant_turns": {
    "1": {"verbosity": 0.25},
    "3": {"warmth": 0.70, "directness": 0.55}
  }
}
```
This field is optional; keep it absent unless you need it.
---
5) Practical constraints for naturalness
Avoid sudden jumps (example: formality 0.2 to 0.9 in one turn).
Keep the same “voice” within a conversation; drift should be subtle.
Use fewer lists at low formality; allow lists when user requests structure.

Profanity denylist format + masking detection rules
This page specifies how to represent and maintain the no-swearing rule as data, plus how to detect masked/obfuscated profanity during filtering.
---
1) Denylist data format (recommended)
Keep denylist separate from the training corpus.
Directory layout
```plain
filters/
  profanity/
    en/
      denylist_v1.txt
      allowlist_v1.txt
      patterns_v1.regex
      README.md
```
`denylist_v1.txt`
One term per line, lowercase.
Terms are whole words unless a pattern file handles obfuscation.
Do not include slurs or explicit terms inside dataset pages if you do not want them visible; store them only in the filter files.
`allowlist_v1.txt`
Words/substrings that can trigger false positives (example: “assistant”, “class”, “passage”).
Used only to suppress matches where a denylist token appears as a substring.
`patterns_v1.regex`
Regex patterns for masked profanity detection.
Keep patterns minimal and test-driven (see test suite spec).
---
2) Detection passes (in order)
Run three passes on each turn (user and assistant).
Pass A: Exact whole-word dictionary
Tokenize on non-letters (keep apostrophes inside words).
Match tokens against denylist.
Reason code: `contains_profanity`
Pass B: Masked/obfuscated detection
Create a detection-only normalized string:
lowercase
unicode NFKC
collapse whitespace
map leetspeak substitutions (example: @ to a, 0 to o, $ to s)
remove repeated punctuation between letters (example: `f**u..n` becomes `fun` in detection pass only)
collapse spaced-out letters (example: `f u c k` becomes `fuck` in detection pass)
Then run regex patterns.
Reason code: `masked_profanity`
Pass C: Contextual false-positive suppression
If a match is only a substring inside an allowlisted word, do not flag.
Reason code: none (suppressed)
---
3) False-positive mitigation rules
Default to whole-word matching.
For masked patterns, require strong evidence:
at least 2 letters in order plus mask characters between them, or
spaced-out letters with consistent single-letter tokens.
Maintain allowlist and unit tests.
---
4) How to record filter results in JSONL
On fail:
```json
"quality": {
  "hard_filtered": true,
  "filter_reasons": ["contains_profanity"],
  "naturalness_score": null,
  "repetition_score": null,
  "helpfulness_score": null
}
```
---
5) Versioning
Bump denylist version when changing meaningfully.
Record filter version in `provenance` if you want full lineage:
`provenance.filter_version`: "profanity/en/v1"

Synthetic persona schema (user + assistant roles)
This page defines a lightweight persona schema for generating synthetic conversations that feel human, while remaining safe (English only, no swearing).
---
1) Why personas
Personas prevent mode collapse (every conversation sounds the same) and help maintain consistency across turns.
---
2) User persona schema
Store under `labels.user_persona` (or `persona.user`).
Recommended fields:
`age_range`: one of {"teen", "20s", "30s", "40s", "50+"}
`chat_style`: {"short", "mixed", "long"}
`patience`: float 0–1 (low means impatient)
`context_richness`: float 0–1 (low means vague)
`directness`: float 0–1
`warmth`: float 0–1
`typical_topics`: short list of topic buckets
`constraints_style`: {"none", "time_pressure", "budget", "schedule"}
Example:

Human-sounding rubric (annotation guide)
This page defines a simple 1–5 scoring rubric to evaluate whether synthetic conversations sound human, while remaining helpful and safe (English only, no swearing).
---
Dimensions (score 1–5)
1) Naturalness
1: stiff, formal, template-like, reads like a corporate bot
3: mostly natural, occasional assistant-like phrasing
5: reads like a real person texting or chatting, varied cadence
2) Coherence
1: contradicts itself, loses thread
3: mostly consistent, minor slips
5: consistent facts, smooth follow-ups, remembers context
3) Concision
1: too long, over-explains
3: sometimes long, sometimes fine
5: matches user’s desired length, no padding
4) Helpfulness
1: vague, non-actionable
3: somewhat useful
5: directly addresses request, practical next step
5) Tone fit (dynamic tone)
1: mismatched (too upbeat, too cold, too formal)
3: acceptable
5: mirrors user energy without being cringe
---
Example signals
High naturalness:
short acknowledgements (“got it”, “yeah that makes sense”) used sparingly
asks 0 or 1 clarifying question when needed
uses contractions naturally
Low naturalness:
“Certainly!” “Moreover…” “In conclusion…”
restating the question
long lists for simple asks
---
Storage
Store scores in JSONL:
```json
"quality": {
  "naturalness_score": 0.82,
  "helpfulness_score": 0.77,
  "repetition_score": 0.12
}
```
If using 1–5 human labels, store under:
`labels.human_ratings`: { naturalness: 4, ... }

Repetition metrics and thresholds
This page defines repetition metrics for filtering and scoring. Repetition is a major driver of “robotic” feel.
---
1) Metrics (assistant text)
Compute over assistant turns only.
A) Repeated n-gram rate
Count repeated 4-grams (or 3-grams for short replies).
`repetition_score` can be normalized to 0–1.
B) Distinct-n
`distinct_1 = unique_unigrams / total_unigrams`
`distinct_2 = unique_bigrams / total_bigrams`
Low distinct-n indicates templating.
C) Line-level repetition
Detect repeated lines or near-duplicates within the same assistant message.
---
2) Thresholds (starter)
Hard filter (drop) if:
repeated 4-gram rate > 0.20 on messages longer than 120 tokens
Soft penalty if:
repeated 4-gram rate in 0.12–0.20
distinct-2 < 0.60 (tune by dataset)
---
3) Storage in JSONL
Store:
`quality.repetition_score` (0–1, higher = more repetitive)
optional: `quality.distinct_1`, `quality.distinct_2`
---
4) Notes
Do not punish legitimate repetition in: short confirmations, scheduling constraints.
Apply different thresholds by `labels.interaction_type` and `labels.cadence`.

Preference-pair spec (robotic vs natural) for later tuning
This page defines how to store preference pairs that teach the model to prefer natural phrasing over stiff phrasing, while keeping the answer equally correct.
---
1) Record format (JSONL)
One line per pair:
```json
{
  "id": "pair-000001",
  "language": "en",
  "prompt": [{"role":"user","text":"…"}],
  "candidates": {
    "robotic": "…",
    "natural": "…"
  },
  "preferred": "natural",
  "labels": {
    "reason": "less_formal|less_paraphrase|better_cadence|fewer_lists",
    "topic_bucket": "…"
  },
  "constraints": {"no_swearing": true},
  "provenance": {
    "generator_model": "NAME",
    "prompt_version": "v1",
    "seed": 0,
    "created_at": "2026-05-16T21:50:00Z"
  }
}
```
---
2) How to generate candidates
Robotic-but-correct candidate
more formal
restates the user prompt
uses heavy structure
longer than needed
Natural-but-correct candidate
minimal acknowledgement
direct answer
short paragraphs
asks at most 1 clarifying question
---
3) Validation rules
Both candidates must be correct.
Natural candidate must not introduce new facts.
No swearing.
---
4) Example pair

Generator prompt scaffolding spec (produce synthetic conversations)
This page defines the prompt scaffolding for the synthetic data generator. It is designed to produce natural English conversations that match the dataset schema and constraints (no swearing).
---
1) Inputs the generator receives
For each record, the generator should be given:
`template_id` (T01…)
`topic_bucket`
`channel`
`user_persona`
`assistant_persona`
baseline `style` values
any drift instructions
hard constraints (English only, no swearing)
max turns / max assistant tokens
---
2) System instruction block (scaffolding)
Use a stable instruction block for the generator model:
Output must be valid JSON matching the canonical schema.
English only.
No swearing.
Conversation must sound like real chat.
Avoid assistant-isms: identity disclaimers, stiff openers, over-structuring.
Follow the selected template constraints (clarifying question count, cadence).
---
3) Template injection block
Provide the template as explicit constraints:
turn pattern
allowed question count
allowed list usage
expected cadence
---
4) Style injection block
Provide numeric style values and drift guidance:
baseline: formality/warmth/directness/verbosity
per-turn drift max ±0.10
mirror user brevity and energy
---
5) Persona injection block
Provide compact persona objects (user and assistant) and tell the generator:
keep persona consistent
keep named entities consistent
---
6) Output rules
Must output one JSON object.
Must include at least 2 turns (user then assistant).
Roles must be correct.
No trailing commentary outside JSON.
---
7) Example meta-prompt (pseudo)
This is a sketch you can implement:
SYSTEM: generator instructions.
DEVELOPER: canonical schema, hard constraints.
USER: template_id, topic_bucket, channel, personas, style values, max turns.
The generator returns one JSON record.
Regenerate-on-fail policy (filters and invalid JSON)
This page defines the regeneration policy when generation fails hard filters or schema validation.
---
1) Failure types
A) Invalid JSON
B) Missing required fields
C) Profanity detected (contains or masked)
D) Unsafe topic drift
E) Hard repetition threshold exceeded
---
2) Retry strategy
For each record:
Max retries: 2
If fail reason is profanity:
resample topic bucket
or reduce slang/humor bias
If fail reason is invalid JSON:
re-run with stricter “JSON only” instruction
---
3) When to drop
Drop the record if:
it fails all retries
it fails with the same reason code repeatedly
Store drop stats per shard (sidecar metadata).
---
4) Recording failures
If you keep an errors log, store:
shard
row
seed
failure reason code
generator model and prompt version
Do not store the profanity token itself in logs.
Allowlist strategy to prevent profanity false positives
This page defines an allowlist approach to reduce profanity false positives.
---
1) Why allowlists
Substring matches can falsely flag safe words (example patterns like “ass” inside “assistant”).
---
2) Rules
Prefer whole-word matching for denylist.
Apply allowlist only for substring detections.
If a denylist hit occurs as a full token, do not suppress.
---
3) Allowlist categories
common safe words that contain risky substrings
technical terms
names
---
4) Testing
Every allowlist entry should have unit tests:
a negative case that must not be flagged
a positive case that must still be flagged
Normalization rules for hashing and deduplication
This page defines normalization rules used to compute deterministic hashes for de-duplication.
---
1) Conversation-level normalization
Input: all turns in order.
Steps:
lowercase
unicode NFKC
trim
collapse all whitespace runs to single spaces
normalize quotes to straight quotes
remove leading/trailing punctuation runs
Canonical concatenation:
each turn as: `role: text` on a new line
Hash:
sha256 of UTF-8 bytes
---
2) Assistant-only normalization
Same normalization, but include assistant turns only.
---
3) Storage
Store in index DB table `hashes`:
`hash_type`: conversation_norm | assistant_norm
`hash_value`: hex string
---
4) Dedup policy
Drop exact duplicates in `filtered/`.
Track duplicate counts per shard.
Profanity filter test suite spec (unit tests)
This page specifies a unit test suite for the profanity filter.
---
1) Test structure
Each test case:
`text`
`expected_flag`: true|false
`expected_reason`: contains_profanity|masked_profanity|null
---
2) Positive cases
exact denylist word as whole token
spaced letters forming a denylist word
leetspeak substitution patterns
punctuation inserted between letters
---
3) Negative cases
allowlisted safe words with risky substrings
similar-looking words that are not profanity
punctuation-heavy but harmless text
---
4) Coverage target
at least 200 tests
include regression tests for every false positive found
---
5) Output
tests should be runnable in CI
failing tests block filter changes
Dataset QA queries (for the index DB)
This page lists example SQL queries to audit dataset health.
---
Filter rate by shard
```sql
SELECT shard_index, COUNT(*) AS total,
  SUM(CASE WHEN hard_filtered=1 THEN 1 ELSE 0 END) AS filtered
FROM conversations c
JOIN filters f ON f.conversation_id = c.id
GROUP BY shard_index
ORDER BY shard_index;
```
Top filter reasons
```sql
SELECT reasons, COUNT(*)
FROM filters
WHERE hard_filtered=1
GROUP BY reasons
ORDER BY COUNT(*) DESC;
```
Naturalness score distribution
```sql
SELECT
  AVG(naturalness) AS avg_nat,
  MIN(naturalness) AS min_nat,
  MAX(naturalness) AS max_nat
FROM scores;
```
Duplication rate
```sql
SELECT hash_value, COUNT(*) AS n
FROM hashes
WHERE hash_type='assistant_norm'
GROUP BY hash_value
HAVING COUNT(*) > 1
ORDER BY n DESC
LIMIT 50;
```
Long-context bucket size
```sql
SELECT COUNT(*)
FROM conversations
WHERE turn_count >= 25 OR char_count >= 12000;
```
---
Notes
Adjust thresholds and bucket definitions as the dataset evolves.
Scoring calibration and threshold selection process
This page defines how to calibrate quality score thresholds without crushing diversity.
---
1) Sample for manual review
Randomly sample 200 records across shards.
Ensure each interaction_type appears.
---
2) Pick initial thresholds

Data generation manifest spec and shard naming conventions
This page specifies shard naming, shard sizing, and manifest file formats.
---
1) Shard sizing
Choose fixed records per shard.
Starter:
100,000 conversations per shard
Rollover:
close shard when record count reached or file exceeds size threshold.
---
2) Shard naming convention
`shard-{NNNNNN}.jsonl` where N is zero-padded.
Examples:
`shard-000000.jsonl`
`shard-000001.jsonl`
Directory convention:
`raw/`, `filtered/`, `scored/`
---
3) Manifest files
Manifests list shard paths relative to dataset root.
`manifests/train.txt`
```plain
filtered/shard-000000.jsonl
filtered/shard-000001.jsonl
...
```
`manifests/val.txt`
Same format.
`manifests/test.txt`
Same format.
---
4) Deterministic splitting
Assign split per record using hash(id):
train: 98%
val: 1%
test: 1%
Store in record:
`labels.split`: train|val|test
---
5) Reproducibility requirements
Each record must include:
`provenance.generator_model`
`provenance.generator_prompt_version`
`provenance.seed`
`provenance.created_at`
`provenance.shard`
`provenance.row`
